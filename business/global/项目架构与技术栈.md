## 分层架构

项目采用经典的三层分离架构：

```
┌─────────────────────────────────────────────────────────┐
│                    Client (HTTP Client)                 │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Controller Layer (src/controllers/)                    │
│  - HTTP 请求处理                                         │
│  - 参数验证                                              │
│  - 响应格式化                                            │
│  - 错误码映射                                            │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Service Layer (src/services/)                          │
│  - 业务逻辑                                              │
│  - 数据库操作                                            │
│  - 事务管理                                              │
│  - 软删除处理                                            │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Data Layer (src/db/schema/)                            │
│  - 表结构定义                                            │
│  - 数据模型                                              │
│  - 公共字段插件                                          │
└─────────────────────────────────────────────────────────┘
```

### 各层职责

#### Controller 层
- **职责**: 处理 HTTP 请求和响应
- **文件**: `src/controllers/*.controller.ts`
- **核心功能**:
  - 路由定义和参数解析
  - 请求体验证和转换
  - HTTP 状态码管理
  - 错误响应格式化

#### Service 层
- **职责**: 实现业务逻辑和数据访问
- **文件**: `src/services/*.service.ts`
- **核心功能**:
  - 业务规则实现
  - 数据库 CRUD 操作
  - 软删除逻辑处理
  - 数据转换和聚合

#### Schema 层
- **职责**: 定义数据模型和表结构
- **文件**: `src/db/schema/*.ts`
- **核心功能**:
  - 数据库表定义
  - 字段类型和约束
  - 表关系定义
  - 公共字段管理

## 核心设计模式

### 1. 软删除模式

所有数据表支持软删除，数据不会从数据库中物理删除：

```typescript
// 公共字段
{
  created_at: timestamp,   // 创建时间
  updated_at: timestamp,   // 更新时间
  deleted_at: timestamp    // 删除时间 (NULL = 未删除)
}

// 查询时自动过滤
.where(isNull(users.deleted_at))
```

**优势**:
- 数据可恢复
- 保留审计记录
- 支持数据分析

### 2. RESTful API 设计

遵循 REST 设计规范：

| HTTP 方法 | 路径 | 操作 |
|-----------|------|------|
| GET | /api/users | 获取列表 |
| GET | /api/users/:id | 获取单个 |
| POST | /api/users | 创建 |
| PUT | /api/users/:id | 更新 |
| DELETE | /api/users/:id | 删除 |

### 3. 类型安全

全链路类型安全：

```typescript
// TypeScript 类型定义
interface CreateUserInput {
  name: string;
  email: string;
}

// Drizzle Schema 类型推导
type User = typeof usersTable.$inferSelect;
```

### 4. 错误处理规范

统一的错误响应格式：

| 场景 | HTTP 状态码 | 错误消息 |
|------|-------------|----------|
| 无效 ID | 400 | Invalid id |
| 参数缺失 | 400 | {field} is required |
| 资源不存在 | 404 | {Resource} not found |
| 唯一约束冲突 | 409 | {Field} already exists |

### 5. JSDoc 注释规范

所有 API 端点必须包含完整的 JSDoc 注释：

```typescript
/**
 * 获取用户
 *
 * @description 获取指定 ID 的用户信息
 * @route GET /api/users/:id
 * @param {number} id - 用户 ID
 * @returns {Promise<Object>} JSON 响应
 * @throws {404} 当用户不存在时
 * @example
 * // GET /api/users/1
 * // Response: { id: 1, name: "John" }
 */
```

## 开发工作流

### 1. 数据库开发流程

```bash
# 1. 定义 Schema
vim src/db/schema/users.ts

# 2. 推送到数据库
pnpm drizzle-kit push

# 3. (可选) 打开数据库管理界面
pnpm drizzle-kit studio
```

```bash
# 1. 创建 Service 层
vim src/services/user.service.ts

# 2. 创建 Controller 层
vim src/controllers/user.controller.ts

pnpm dev

# 运行所有测试
pnpm test

```

## 技术选型理由

### 为什么选择 Hono？

| 优势 | 说明 |
|------|------|
| **轻量快速** | 基于 Web Standards，极小的包体积 |
| **类型安全** | 原生 TypeScript 支持，完整的类型推导 |
| **中间件生态** | 丰富的官方中间件（CORS、JWT、Validator 等） |
| **跨平台** | 支持 Node.js、Cloudflare Workers、Deno 等 |
| **现代化** | 基于 Web Standards API，不依赖特定运行时 |

### 为什么选择 Drizzle ORM？

| 优势 | 说明 |
|------|------|
| **TypeScript-first** | 完全用 TypeScript 编写，类型推导优秀 |
| **零依赖** | 核心包几乎无依赖，打包体积小 |
| **性能优异** | 比传统 ORM 更快，接近原生 SQL |
| **SQL-like** | 查询语法接近 SQL，学习成本低 |
| **迁移管理** | 提供强大的迁移工具 |

### 为什么选择 PostgreSQL？

| 优势 | 说明 |
|------|------|
| **功能强大** | 支持复杂查询、JSON、索引类型 |
| **数据完整性** | ACID 事务，外键约束 |
| **开源生态** | 活跃的社区和丰富的扩展 |
| **性能优秀** | 适合读密集和写密集场景 |

### 为什么选择 Vitest？

| 优势 | 说明 |
|------|------|
| **极速启动** | 基于 Vite，冷启动快 |
| **原生 ESM** | 无需转换，直接测试 ES 模块 |
| **UI 界面** | 可视化测试界面 |
| **覆盖率内置** | 集成 c8 覆盖率工具 |

## 开发规范

### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | kebab-case | `user.controller.ts` |
| 类名 | PascalCase | `UserService` |
| 函数/变量 | camelCase | `getAllUsers` |
| 常量 | UPPER_SNAKE_CASE | `DATABASE_URL` |
| 表名 | snake_case | `users_table` |
| 数据库字段 | snake_case | `created_at` |

### API 响应规范

**成功响应**:
```json
{
  "id": 1,
  "name": "John",
  "email": "john@example.com"
}
```

**错误响应**:
```json
{
  "message": "User not found"
}
```

## 性能考虑

- 使用索引优化查询性能
- 连接池管理（由 postgres 驱动提供）
- 预编译语句（Drizzle 自动处理）
- 查询结果缓存（可选，未来版本）

## 安全考虑

- SQL 注入防护（使用 ORM 参数化查询）
- XSS 防护（输出转义）
- CORS 配置（生产环境需限制来源）
- 输入验证（Controller 层严格校验）
- 环境变量隔离（敏感信息不进代码库）
