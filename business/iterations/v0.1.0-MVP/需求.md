# 需求文档

> v0.1.0-MVP 版本需求定义

## 迭代概览

**版本**: v0.1.0-MVP
**目标**: 实现核心用户认证功能（注册、登录、令牌管理）
**时间**: 预估 1 周
**状态**: 📝 规划中

## 背景和动机

### 业务背景
本项目旨在打造一个极简、安全、可扩展的用户认证模板，供开发者快速启动新项目。当前项目已具备基础的用户 CRUD 功能，但缺少核心的认证能力。

### 用户痛点
- 开发者需要在每个项目中重新实现用户认证逻辑
- 现有方案过于复杂或不够安全
- 缺少统一的认证标准和最佳实践

### 解决方案概述
本版本将实现基于 JWT 的用户认证系统，包括用户注册、登录、令牌刷新和注销功能，为后续功能扩展奠定基础。

## 用户故事

### 故事 1: 开发者需要用户注册功能

**描述**: 作为开发者，我需要用户注册功能，以便新用户可以创建账户并开始使用系统。

**验收标准**:
- [ ] 用户可以通过邮箱和密码注册
- [ ] 密码使用 bcrypt 哈希存储（成本因子 12）
- [ ] 邮箱必须唯一，重复邮箱返回明确错误
- [ ] 注册成功返回用户信息（不含密码）
- [ ] 输入参数有完整验证

**优先级**: 🔴 高

### 故事 2: 用户需要登录并获取访问令牌

**描述**: 作为用户，我需要使用邮箱和密码登录，以便获取访问令牌并访问受保护的资源。

**验收标准**:
- [ ] 用户通过邮箱和密码登录
- [ ] 登录成功返回 JWT 访问令牌（15分钟有效期）
- [ ] 登录成功返回刷新令牌（7天有效期）
- [ ] 登录失败返回明确的错误信息
- [ ] 错误的邮箱或密码不泄露具体错误（统一提示）

**优先级**: 🔴 高

### 故事 3: 用户需要刷新访问令牌

**描述**: 作为用户，我需要使用刷新令牌获取新的访问令牌，以便在访问令牌过期后继续使用系统而无需重新登录。

**验收标准**:
- [ ] 使用刷新令牌获取新的访问令牌
- [ ] 刷新令牌验证有效性（存在、未过期）
- [ ] 刷新令牌轮换（生成新的刷新令牌）
- [ ] 无效的刷新令牌返回明确错误

**优先级**: 🔴 高

### 故事 4: 用户需要注销

**描述**: 作为用户，我需要注销登录，以便使当前会话失效。

**验收标准**:
- [ ] 注销时删除服务端的刷新令牌
- [ ] 注销成功返回确认消息
- [ ] 重复注销返回成功（幂等）

**优先级**: 🔴 高

### 故事 5: 用户需要获取自己的信息

**描述**: 作为用户，我需要获取自己的个人信息，以便系统可以识别当前登录用户。

**验收标准**:
- [ ] 使用 JWT 访问令牌获取当前用户信息
- [ ] 无效或过期的令牌返回 401
- [ ] 返回用户信息不含敏感字段（密码哈希）

**优先级**: 🔴 高

## 功能列表

### 核心功能 (MUST HAVE)

#### F1: 用户注册
**描述**: 用户使用邮箱和密码创建账户
**用户价值**: 允许新用户加入系统
**技术复杂度**: 中
**依赖项**: users 表、密码哈希

#### F2: 用户登录
**描述**: 验证用户身份并签发令牌
**用户价值**: 用户可以访问受保护的资源
**技术复杂度**: 中
**依赖项**: users 表、JWT 签发、sessions 表

#### F3: 令牌刷新
**描述**: 使用刷新令牌获取新的访问令牌
**用户价值**: 无需频繁重新登录
**技术复杂度**: 中
**依赖项**: sessions 表、JWT 验证

#### F4: 注销
**描述**: 删除服务端会话
**用户价值**: 用户可以安全退出
**技术复杂度**: 低
**依赖项**: sessions 表

#### F5: 获取当前用户
**描述**: 获取当前登录用户的信息
**用户价值**: 系统识别用户身份
**技术复杂度**: 低
**依赖项**: JWT 验证中间件

### 增强功能 (SHOULD HAVE)

#### F6: 密码强度验证
**描述**: 验证密码强度（长度、复杂性）
**用户价值**: 提高账户安全性
**技术复杂度**: 低

#### F7: 邮箱格式验证
**描述**: 验证邮箱格式的有效性
**用户价值**: 减少无效注册
**技术复杂度**: 低

### 未来功能 (NICE TO HAVE)

#### F8: 多设备登录管理
**描述**: 查看和管理所有登录设备
**用户价值**: 更好的账户安全控制
**技术复杂度**: 高

#### F9: 记住我功能
**描述**: 延长刷新令牌有效期
**用户价值**: 减少登录频率
**技术复杂度**: 中

## 非功能性需求

### 性能要求
- API 响应时间: < 200ms (P95)
- 数据库查询时间: < 100ms (P95)
- JWT 验证时间: < 10ms
- bcrypt 哈希时间: < 500ms

### 安全要求
- 密码必须使用 bcrypt 哈希（成本因子 12）
- JWT 使用环境变量配置的密钥签名
- 刷新令牌存储哈希值而非明文
- 登录接口速率限制（5 次/分钟/IP）
- 所有用户输入必须验证
- SQL 注入防护（使用 ORM）

### 可用性要求
- 系统可用性: 99%
- 数据持久化: 是
- 错误处理: 友好的错误消息
- API 响应格式统一

## 依赖项

### 外部依赖
- PostgreSQL 数据库

### 技术依赖
- Hono ^4.12.2
- Drizzle ORM ^0.45.1
- bcrypt 或 bcryptjs (密码哈希)
- jsonwebtoken 或 @hono/jwt (JWT 签发和验证)

## 风险与问题

| 风险/问题 | 影响 | 概率 | 缓解措施 |
|----------|------|------|----------|
| JWT 密钥泄露 | 高 | 中 | 使用强密钥，存储在环境变量中 |
| 刷新令牌泄露 | 高 | 中 | 存储哈希值，设置合理过期时间 |
| bcrypt 性能影响 | 中 | 低 | 异步处理，必要时调整成本因子 |
| 并发登录冲突 | 低 | 低 | 允许多设备登录，后续版本管理 |

## 兼容性说明

### 与现有版本的兼容性
- **需要修改**: users 表结构（添加 password_hash、is_admin）
- **新建表**: sessions 表
- **迁移策略**: 添加字段并设置默认值

### 数据迁移需求
```sql
-- 添加新字段（带默认值）
ALTER TABLE users ADD COLUMN password_hash VARCHAR(255);
ALTER TABLE users ADD COLUMN is_admin BOOLEAN DEFAULT FALSE NOT NULL;
```

### API 变更
- 新增 `/auth/*` 路由
- 现有 `/api/users/*` 路由保持不变
- 新增 JWT 中间件
