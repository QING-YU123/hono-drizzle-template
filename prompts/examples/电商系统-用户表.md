## 📋 场景说明

这是一个电商系统的用户管理模块，包含用户注册、登录、个人信息管理等功能。

**设计思路：**
- 用户表存储核心信息（用户名、邮箱、密码）
- 使用软删除保留数据
- 支持角色区分（普通用户/管理员）
- 记录最后登录时间

### 步骤 1：创建用户表

```
请创建一个 用户 数据表，包含以下字段：

id: integer (主键) [自增]
username: string (用户名) [必填, 唯一]
password: string (密码) [必填]
email: string (邮箱) [必填, 唯一]
phone: string (手机号) [可选]
nickname: string (昵称) [可选]
avatar: string (头像URL) [可选]
gender: string (性别: male/female/other) [可选, 默认: other]
birth_date: date (生日) [可选]
status: string (状态: active/inactive/banned) [必填, 默认: active]
role: string (角色: user/admin) [必填, 默认: user]
last_login_at: timestamp (最后登录时间) [可选]
created_at: timestamp (创建时间) [自动生成]
updated_at: timestamp (更新时间) [自动生成]
deleted_at: timestamp (删除时间) [软删除]

要求：
1. 表名为 users
2. 使用 PostgreSQL 数据类型
3. 包含公共字段：created_at, updated_at, deleted_at
4. 为主键字段添加自增
5. 为 username 和 email 添加唯一约束
6. 文件保存在 src/db/schema/users.ts
7. 创建完成后，使用 drizzle-kit push 推送到数据库
```

### 步骤 2：生成完整 CRUD

```
请为 users 表生成完整的 CRUD API，包括以下功能：

1. Service 层（src/services/user.ts）：
   - create: 创建新用户
   - getAll: 获取所有用户（包含软删除过滤）
   - getById: 根据 ID 获取单个用户
   - getByUsername: 根据用户名获取用户
   - getByEmail: 根据邮箱获取用户
   - update: 更新用户信息
   - updatePassword: 更新用户密码
   - updateLastLogin: 更新最后登录时间
   - delete: 软删除用户（设置 deleted_at）
   - restore: 恢复已删除用户
   - permanentDelete: 永久删除用户

2. Controller 层（src/controllers/user.ts）：
   - POST /users - 创建用户（注册）
   - GET /users - 获取所有用户（管理员）
   - GET /users/:id - 根据 ID 获取用户
   - PUT /users/:id - 更新用户信息
   - PUT /users/:id/password - 修改密码
   - DELETE /users/:id - 软删除用户

要求：
1. 遵循项目的 MVC 分层架构
2. 所有查询都过滤已删除记录（where(isNull(table.deleted_at))）
3. 密码字段不要在查询中返回（安全考虑）
4. 使用数据库实例（从 src/index.ts 导入）
5. 返回统一的 JSON 响应格式
6. 包含适当的错误处理
7. 在 src/index.ts 中注册路由
```

### 步骤 3：添加输入验证

```
请为 用户 的 API 端点添加输入验证。

需要验证的端点：
1. POST /users - 用户注册
2. PUT /users/:id - 更新用户信息
3. PUT /users/:id/password - 修改密码

验证规则：
username: 必填, 字符串, 最小长度3, 最大长度20, 字母数字下划线, 唯一
password: 必填, 字符串, 最小长度6, 最大长度100
email: 必填, 字符串, 邮箱格式, 唯一
phone: 可选, 字符串, 手机号格式（11位数字）
nickname: 可选, 字符串, 最大长度30
avatar: 可选, 字符串, URL格式
gender: 可选, 枚举: male/female/other
birth_date: 可选, 日期格式
status: 可选, 枚举: active/inactive/banned
role: 可选, 枚举: user/admin

要求：
1. 在 Controller 层添加验证逻辑
2. 验证失败返回 400 错误和清晰的错误信息
3. 密码验证要包含强度检查（至少包含字母和数字）
4. 验证通过后再执行业务逻辑
5. 使用 TypeScript 类型确保类型安全
```

### 步骤 4：生成种子数据

```
请为 users 表生成种子数据（Seed Data）。

数据要求：
1. 生成 30 条测试用户记录
2. 包含 1 个管理员账号（username: admin, role: admin）
3. 包含 29 个普通用户
4. 数据要真实可信
5. 包含不同状态的用户（active, inactive, banned）

数据字段：
username: admin, user1, user2, user3...user29
password: 所有用户密码统一为 password123（实际项目中要哈希加密）
email: admin@example.com, user1@example.com, user2@example.com...
phone: 随机生成11位手机号（138****1234格式）
nickname: 随机中文昵称（张三丰、李小龙、王小明...）
avatar: https://api.dicebear.com/7.x/avataaars/svg?seed={username}
gender: 随机 male/female/other
birth_date: 1970-2005年之间的随机日期
status: 80% active, 15% inactive, 5% banned
role: admin为admin，其他为user
last_login_at: 最近30天内随机时间

要求：
1. 创建独立的种子数据脚本文件（src/db/seed/user.ts）
2. 使用数据库实例（从 src/index.ts 导入）
3. 插入前检查数据是否已存在（避免重复插入）
4. 批量插入数据（使用事务）
5. 密码要哈希加密后再存储（使用 bcrypt）
6. 包含成功/失败提示
7. 提供执行命令说明
```

### 步骤 5：添加搜索功能

```
请为 用户 的 GET /users 端点添加搜索和筛选功能。

需要添加的搜索功能：
关键词搜索: username (支持模糊搜索用户名)
关键词搜索: nickname (支持模糊搜索昵称)
关键词搜索: email (支持模糊搜索邮箱)
精确筛选: status (筛选用户状态)
精确筛选: role (筛选用户角色)
精确筛选: gender (筛选性别)
范围查询: created_at (注册日期范围: start_date, end_date)
范围查询: last_login_at (最后登录日期范围)
排序: created_at, username, last_login_at (支持多字段排序)

要求：
1. 支持多条件组合查询
2. 所有搜索参数都是可选的
3. 关键词搜索使用模糊匹配（LIKE，不区分大小写）
4. 筛选使用精确匹配
5. 支持排序（order_by 和 order_direction）
6. 返回匹配的记录总数
7. 保持软删除过滤（where(isNull(table.deleted_at))）
8. 不要返回密码字段
```

### 步骤 6：添加分页功能

```
请为 用户 的 GET /users 端点添加分页功能。

分页参数：
- page: 页码（默认: 1，最小: 1）
- page_size: 每页数量（默认: 20，最小: 1，最大: 100）

返回数据包含：
- data: 用户列表（不包含密码字段）
- total: 总记录数
- page: 当前页码
- page_size: 每页数量
- total_pages: 总页数
- has_next: 是否有下一页
- has_prev: 是否有上一页

要求：
1. 支持分页参数（page, page_size）
2. 计算总记录数和总页数
3. 限制 page_size 的最大值为 100
4. 支持 limit 和 offset 查询
5. 返回完整的分页元数据
6. 可以与搜索功能组合使用
7. 不要返回密码字段
```

### 步骤 7：添加权限控制

```
请为 用户 的 API 添加权限控制功能。

权限要求：
GET /users: 仅管理员（获取所有用户列表）
GET /users/:id: 用户本人或管理员（获取用户信息）
POST /users: 所有用户（用户注册）
PUT /users/:id: 用户本人或管理员（更新用户信息）
PUT /users/:id/password: 用户本人（修改密码）
DELETE /users/:id: 用户本人或管理员（删除账号）

用户角色定义：
guest: 访客（未登录），可以注册账号
user: 普通用户，可以查看和修改自己的信息
admin: 管理员，可以操作所有用户数据

要求：
1. 实现基于角色的访问控制（RBAC）
2. 验证用户是否已登录（认证）
3. 检查用户是否有权限执行操作（授权）
4. 资源所有权验证（用户只能操作自己的资源）
5. 返回适当的 HTTP 状态码（401 未认证, 403 无权限）
6. 为管理员提供特殊权限（可以操作所有资源）
7. 在 Controller 层添加权限检查逻辑
8. 注册端点为公开访问，其他端点需要认证
```

## 💡 使用建议

1. **开发顺序**：按照步骤 1 → 2 → 3 → 4 → 5 → 6 → 7 的顺序执行
2. **测试账号**：种子数据生成的管理员账号可用于测试权限
3. **密码安全**：实际项目中，密码必须使用 bcrypt 或 argon2 加密
4. **Token 认证**：需要额外实现 JWT token 生成和验证
5. **扩展功能**：可添加邮箱验证、密码重置等功能

## ⚠️ 注意事项

1. 密码字段不要在 API 响应中返回
2. 敏感操作（修改密码、删除账号）需要验证当前密码
3. 管理员账号要谨慎创建和保管
4. 用户状态为 banned 时，不允许登录
5. 软删除的用户不应该能够登录
