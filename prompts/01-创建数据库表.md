# 创建数据库表 - 思考指南

## 🎯 这个指南帮助你

创建数据库表之前，你需要想清楚这些问题。**不要跳过思考过程！**

## 🤔 设计之前先问自己

### 1. 这个表存储什么数据？

```
例子：
❌ 太模糊："存储商品信息"
✅ 清晰："存储电商平台的商品信息，包括名称、价格、库存、分类等"
```

### 2. 表应该叫什么名字？

**这是你的决策，参考建议：**

- 用英文（编程惯例）
- 用复数形式（存储多个记录）
- 用下划线分隔多个单词

```
例子：
存储用户 → users
存储产品 → products
存储订单 → orders
存储用户资料 → user_profiles
存储商品分类 → product_categories
```

### 3. 需要哪些字段？

**为每个字段问自己：**

| 问题 | 说明 | 例子 |
|------|------|------|
| 字段叫什么？ | 用英文，见名知意 | `user_name` 比 `n` 好 |
| 什么类型？ | 字符串/数字/日期/布尔 | 价格用 `numeric` 不用 `integer` |
| 必填吗？ | `.notNull()` | 用户名必填，简介可选 |
| 有默认值吗？ | `.default()` | 状态默认 `active` |
| 必须唯一吗？ | `.unique()` | 邮箱、用户名 |
| 需要索引吗？ | 查询性能 | 经常查询的字段 |

### 4. 字段怎么命名？

**命名风格（你自己决定）：**

```
风格1: snake_case（推荐）
user_name, created_at, product_id

风格2: camelCase
userName, createdAt, productId

⚠️ 一旦选择，保持一致！
```

### 5. 需要关系吗？

```
外键关系例子：
- orders.user_id → users.id (订单属于用户)
- products.category_id → categories.id (商品属于分类)
- order_items.order_id → orders.id (明细属于订单)
```

## 📊 设计你的表（在纸上或脑子里）

### 示例：设计一个商品表

**我的业务需求：**
- 电商平台卖商品
- 需要记录商品名称、价格、库存
- 需要知道商品属于哪个分类
- 需要记录商品创建和修改时间
- 删除商品时不能真删除（可能有订单关联）

**我的设计决策：**

| 决策点 | 我的选择 | 为什么 |
|--------|---------|--------|
| 表名 | `products` | 复数，存储多个商品 |
| 主键 | `id`，自增 | 简单标准 |
| 商品名 | `product_name`，必填，唯一 | 每个商品名称不同 |
| 价格 | `price`，必填，decimal(10,2) | 需要精确到分 |
| 库存 | `stock`，默认0 | 整数，可缺货 |
| 分类 | `category_id`，可选 | 关联分类表 |
| 软删除 | `deleted_at` | 保留数据 |

## 💬 向 AI 清楚表达你的设计

**好的提问方式：**

```
我想创建一个商品表，具体设计如下：

表名：products
字段设计：
- id: 主键，自增
- product_name: 商品名称，必填，唯一，varchar(255)
- price: 价格，必填，numeric(10,2)
- stock: 库存数量，整数，默认0
- category_id: 分类ID，可选，关联分类表
- description: 商品描述，可选，text
- created_at: 创建时间
- updated_at: 更新时间
- deleted_at: 删除时间（软删除）

请用 Drizzle ORM 实现，使用 PostgreSQL。
```

**为什么这样问更好：**
- ✅ 你明确说明了每个字段的细节
- ✅ 你决定了命名（不是AI决定）
- ✅ 你说明了业务需求（为什么这样设计）
- ✅ AI 只负责实现，不负责设计

## 📚 技术参考（不是模板）

这些是**技术选项**，你根据需求选择：

### 数据类型选择

| 存储内容 | 可选类型 | 如何选择 |
|---------|---------|---------|
| 短文本 | `varchar(n)` | n是最大长度，如用户名50 |
| 长文本 | `text` | 文章、描述等不限长度 |
| 整数 | `integer` | 年龄、数量 |
| 大整数 | `bigint` | 可能很大的ID |
| 精确小数 | `numeric(p,s)` | 价格（p是总位数，s是小数位） |
| 普通小数 | `real` | 不需要精确的数值 |
| 布尔值 | `boolean` | 是/否 |
| 日期时间 | `timestamp` | 创建时间、更新时间 |
| JSON数据 | `jsonb` | 灵活的配置、元数据 |

### 约束选项

| 约束 | Drizzle语法 | 何时使用 |
|------|-------------|---------|
| 主键 | `.primaryKey()` | 每个表的唯一标识 |
| 不允许空 | `.notNull()` | 必填字段 |
| 唯一 | `.unique()` | 不能重复的值（邮箱、用户名） |
| 默认值 | `.default(value)` | 有默认值的字段 |
| 自增 | `.generatedAlwaysAsIdentity()` | 主键自动增长 |

### 时间字段（本项目约定）

```typescript
// 从 columns.helpers.ts 导入
import { createdAt, updatedAt, deletedAt } from './plugin/columns.helpers';

// 使用
...createdAt,    // created_at timestamp, 自动生成
...updatedAt,    // updated_at timestamp, 自动生成
...deletedAt,    // deleted_at timestamp, 可选（软删除）
```

## ⚠️ 常见错误

### 错误1：让AI猜测
```
❌ "帮我创建一个商品表"
   AI会猜测你要什么，可能不符合你的需求
```

### 错误2：复制别人的设计
```
❌ 照抄教程或别人的表结构
   你的业务可能不一样，字段需求也不同
```

### 错误3：不考虑未来
```
❌ 只想当前需求，不考虑扩展
   比如：用户表没有考虑是否需要支持多种登录方式
```

### 错误4：命名不一致
```
❌ 一会儿用 user_name，一会儿用 userName
   选择一种风格，保持一致
```

- [ ] 我决定了每个字段的名字
- [ ] 我选择了合适的类型
- [ ] 我考虑了哪些字段必填
- [ ] 我考虑了哪些字段需要唯一约束

- [ ] 我理解了为什么这样设计
