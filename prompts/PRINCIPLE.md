# 重要原则：用户保持完全控制

## 🎯 核心原则

**用户必须保持对项目的完全控制权和设计权。**

AI 不应该替用户做设计决策，而是应该：
1. **理解用户的设计**
2. **帮助用户实现**
3. **解释技术选项**
4. **回答技术问题**

## ❌ 当前问题

### 技能文件（Skills）的问题

**文件位置**：`.claude/skills/*.md`

**问题**：
- 技能文件规定了固定的函数命名（`getAll[Resource]`, `create[Resource]`）
- 技能文件规定了固定的代码模式
- 用户无法决定自己的变量名、函数名、类名
- AI 自动生成代码，用户失去控制

**示例（错误）**：
```typescript
// 技能文件规定必须这样命名
async getAll[Resource]() { ... }
async create[Resource](data) { ... }
async update[Resource](id, data) { ... }
```

**应该是**：
```typescript
// 用户决定怎么命名
async findAllProducts() { ... }  // 用户喜欢这个命名
async createProduct(data) { ... }  // 用户决定
async updateProduct(id, data) { ... }  // 用户控制
```

### 提示词模板（Prompts）的问题

**文件位置**：`prompts/*.md`

**问题**：
- 提示词是"复制粘贴模板"
- 用占位符让用户替换
- 用户不需要思考，只需要替换
- 违背了"让用户设计"的原则

**示例（错误）**：
```
请创建一个 [资源名称] 数据表，包含以下字段：
[字段列表]

用户只需要：
[资源名称] → 商品
[字段列表] → name, price, stock
```

**应该是**：
```
思考指南帮助用户理解：
1. 我需要什么字段？
2. 字段叫什么名字？
3. 什么数据类型？
4. 为什么要这样设计？

然后用户用自然语言告诉AI：
"我想创建一个商品表，字段包括..."
```

## ✅ 应该怎么改

### 技能文件应该：

1. **询问用户，不要假设**
   ```
   ❌ "创建名为 getAll[Resource] 的函数"
   ✅ "你想给这个函数起什么名字？它要做什么？"
   ```

2. **提供选项，不要规定**
   ```
   ❌ "必须使用 soft delete 模式"
   ✅ "你想使用软删除还是硬删除？软删除保留数据，硬删除永久删除"
   ```

3. **解释原因，不要命令**
   ```
   ❌ "必须这样做"
   ✅ "这样做的好处是...，你也可以选择..."
   ```

### 提示词模板应该：

1. **从"模板"改为"思考指南"**
   - 帮助用户理解要考虑什么
   - 引导用户自己做决策
   - 展示如何向AI表达设计

2. **从"复制粘贴"改为"理解+表达"**
   - 不提供可复制的模板
   - 提供思考框架
   - 提供技术参考

3. **从"占位符"改为"决策点"**
   - 不用 [表名] 这样的占位符
   - 用"你决定表名应该叫什么"
   - 让用户思考为什么要这样命名

## 🔄 修改原则

### 对于所有提示词文件：

```
旧结构：
📋 使用说明
📝 复制以下内容（模板）
🔧 如何修改（替换占位符）

新结构：
🎯 这个指南帮助你（理解目的）
🤔 设计之前先问自己（引导思考）
📊 设计你的方案（自己做决策）
💬 向AI表达（说清楚想法）
📚 技术参考（可选方案）
⚠️ 常见错误（避免什么）
✅ 检查清单（确保思考过）
```

### 对于所有技能文件：

```
旧模式：
用户 → AI自动决策 → AI生成代码

新模式：
用户 → AI询问设计 → 用户做决策 → AI实现
```

## 📚 需要添加的内容

### 每个指南都应该有：

1. **为什么需要思考**
   ```
   不直接给答案，而是说明为什么这个设计很重要
   ```

2. **决策框架**
   ```
   给出问题列表，用户回答这些问题就是设计过程
   ```

3. **示例对话**
   ```
   展示好的提问方式和不好的提问方式
   ```

4. **技术选项**
   ```
   列出可能的技术方案，让用户选择
   不是推荐"最佳方案"，而是说明"各种方案的特点"
   ```

## 🎯 验证标准

修改后，用户应该能够：

- [ ] 理解为什么要做某个设计
- [ ] 自己决定变量名、函数名、类名
- [ ] 向AI清楚表达自己的设计
- [ ] 理解AI生成的每一行代码
- [ ] 知道如何修改不符合预期的代码
- [ ] 感觉自己是项目的主人，而不是AI的助手

## ⚠️ 危险信号检查

如果用户出现以下行为，说明修改不够：

- 复制提示词只替换占位符
- 不理解生成的代码
- 不知道为什么用某个函数名
- 无法解释自己的设计决策
- 感觉"AI帮我写完了"

## 📖 推荐阅读

这种理念类似于：
- "Teach a man to fish" vs "Give a man a fish"
- 软件工程中的"Trade-off"思维
- 敏捷开发中的"Individuals and interactions over processes and tools"
