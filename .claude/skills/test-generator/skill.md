# API 测试生成器

这个技能帮助你根据版本测试用例文档自动生成 Vitest 测试代码，运行测试并将测试报告保存到对应版本的测试报告目录中。

## 🎯 技能概述

### 主要功能
- 📋 **版本管理**：识别当前项目版本，获取对应版本的测试用例
- 📖 **读取测试用例**：从 `business/iterations/{version}/测试/测试用例.md` 读取测试用例
- ✨ **生成测试代码**：自动生成符合项目规范的 Vitest 测试代码
- 🚀 **运行测试**：执行测试并收集结果
- 📊 **生成测试报告**：将测试报告保存到 `business/iterations/{version}/测试/测试报告/`

### 前提条件
- ✅ 项目已配置 Vitest 环境
- ✅ 依赖已安装（`pnpm install`）
- ✅ 测试用例文档存在于对应版本的目录中
- ✅ Hono 应用可被导入用于测试

## 🔄 完整工作流程

### 第一阶段：版本识别和环境分析（必做）

#### 1. 识别当前版本
```bash
# 扫描 business/iterations 目录，识别所有版本
Glob business/iterations/*

# 与用户确认当前要测试的版本
# 如果只有一个版本，自动使用该版本
# 如果有多个版本，询问用户选择
```

**版本识别规则：**
- 默认选择最新的版本（按字母/数字排序）
- 版本格式：`v{major}.{minor}.{patch}-{tag}`（如 `v0.1.0-MVP`）
- 确认版本目录下存在 `测试/测试用例.md` 文件

#### 2. 分析测试环境
```bash
# 检查 Vitest 配置
Read tests/vitest.config.ts
Read package.json  # 查看 test 脚本

# 检查测试目录结构
Glob tests/**/*.test.ts

# 了解应用入口
Read src/index.ts
```

**分析要点：**
- Vitest 配置和测试环境设置
- 现有测试文件的命名和结构
- 应用的导入方式（用于测试中模拟请求）
- 数据库测试环境设置（如果使用）

### 第二阶段：读取测试用例

#### 1. 定位测试用例文档
```bash
# 根据版本确定测试用例路径

### 第三阶段：需求确认

#### 必须确认的内容

   - 测试用例文档路径

2. **测试范围**
   - 要测试哪些模块/接口
   - 是否需要生成所有接口的测试
   - 是否跳过某些测试用例

3. **特殊测试需求**
   - 是否需要身份认证
   - 是否需要数据库测试环境

4. **测试输出**
   - 测试文件输出位置（默认 `tests/api/`）

### 第四阶段：生成测试代码

#### 1. 测试文件命名
- 格式：`[module-name].test.ts`

- 示例：`users.test.ts`, `articles.test.ts`

#### 2. 测试代码生成规则

**基于详细表格格式生成：**
```typescript
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import { app } from '../../src/index';

describe('用户 API 测试', () => {
  describe('GET /api/users', () => {
    it('UC-GET-001: 成功获取用户列表（存在用户）', async () => {
      // 根据前置条件准备数据
      // 根据测试步骤发送请求
      const res = await app.request('/api/users', { method: 'GET' });

      // 根据预期结果验证
      expect(res.status).toBe(200);
      const body = await res.json();
      expect(Array.isArray(body)).toBe(true);
      // 更多断言...
    });

    it('UC-GET-002: 成功获取用户列表（无用户）', async () => {
      const res = await app.request('/api/users', { method: 'GET' });
      expect(res.status).toBe(200);
      const body = await res.json();
      expect(body).toEqual([]);
    });
  });
});
```

**基于简化表格格式生成：**
```typescript
describe('POST /api/users', () => {
  it('should create user with valid data', async () => {
    const res = await app.request('/api/users', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ name: '张三', email: 'test@example.com' })
    });

    expect(res.status).toBe(201);
    const body = await res.json();
    expect(body).toHaveProperty('id');
    expect(body).toHaveProperty('name', '张三');
  });
});
```

### 1. UC-POST-007: 失败：邮箱重复（唯一性约束）

**测试内容**: 验证邮箱唯一性约束
**预期结果**: 响应状态码为 409，包含错误信息
**实际结果**: 响应状态码为 201

**错误分析**:
```
expected 201 to be 409
```

**问题诊断**:
- 数据库可能没有设置 email 字段的唯一约束
- 或者业务逻辑没有正确检查邮箱重复

**修复建议**:
1. 检查 `src/db/schema/users.ts` 中 email 字段是否有 `.unique()` 约束

3. 运行 `pnpm drizzle-kit push` 确保数据库约束生效

### 2. UC-PUT-011: 失败：更新邮箱与其他用户冲突

**测试内容**: 验证更新邮箱时的唯一性约束
**预期结果**: 响应状态码为 409
**实际结果**: 响应状态码为 200

**问题诊断**:
- 更新操作没有检查邮箱是否与其他用户冲突

**修复建议**:
- 在用户更新逻辑中添加邮箱冲突检查
- 查询数据库确认新邮箱是否已被其他用户使用

## 🚨 常见问题处理

### 问题 1：找不到测试用例文档
