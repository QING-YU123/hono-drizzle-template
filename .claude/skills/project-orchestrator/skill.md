# 项目流程编排器

这个技能是整个项目开发的"大脑"，负责协调所有其他 skills，根据项目当前状态智能决策下一步行动，记录执行进度，实现从版本设计到代码审查的完整自动化流程。

## 🎯 技能概述

### 主要功能
- 🗺️ **流程管理**：管理完整的项目开发流程（设计→开发→测试→审查）
- 📊 **状态追踪**：实时追踪项目状态和进度
- 🧠 **智能决策**：根据当前状态和用户需求决定下一步行动
- 📝 **进度记录**：维护详细的项目执行日志
- 🔄 **技能协调**：指挥和协调其他 skills 的执行

### 管理的开发流程

```
📋 需求分析阶段
   ↓
🎨 版本设计阶段 (version-designer)
   ↓
🗄️ 数据库设计阶段 (schema-generator)
   ↓
💻 业务开发阶段 (business-generator)
   ↓
📚 API文档生成阶段 (openapi-generator)
   ↓
🧪 测试设计阶段 (test-case-designer)
   ↓
✅ 测试执行阶段 (test-generator)
   ↓
🔍 代码审查阶段 (code-reviewer)
   ↓
🎉 版本完成
```

### 前提条件
- ✅ 项目已初始化（包含 `business/` 目录结构）
- ✅ 所有相关 skills 已安装
- ✅ 项目配置文件完整（package.json, tsconfig.json 等）

### 第二阶段：智能决策和流程规划

#### 1. 理解用户需求
与用户交流，明确：
- **目标**：用户想要做什么？
  - 开始新版本开发？
  - 继续当前版本的某个阶段？
  - 查看项目进度？
  - 修复某个问题？

- **范围**：涉及哪些功能或模块？
- **优先级**：紧急程度如何？
- **约束**：有什么时间或资源限制？

#### 2. 决策树

```markdown
## 智能决策逻辑

### 场景 1：用户想要开始新版本
**条件**：
- 用户表达"需要新功能"、"要开发新版本"等意图
- 当前没有活跃的版本开发

**决策**：
1. 调用 `/version-designer` 进行版本设计
2. 等待用户确认设计方案
3. 创建新的工作流实例

### 场景 2：用户想要继续当前版本
**条件**：
- 存在活跃的版本开发
- 用户表达"继续"、"下一步"等意图

**决策**：
根据当前阶段决定：
- `design` → 调用 `/schema-generator`
- `schema` → 调用 `/business-generator`
- `development` → 调用 `/openapi-generator`
- `documentation` → 调用 `/test-case-designer`
- `test-design` → 调用 `/test-generator`
- `testing` → 调用 `/code-reviewer`

### 场景 3：用户想要跳到特定阶段
**条件**：
- 用户明确指定要执行某个阶段

**决策**：
1. 检查前置条件是否满足
2. 如果满足，直接调用对应的 skill
3. 如果不满足，提示用户先完成前置步骤

### 场景 4：用户想要查看进度
**条件**：
- 用户表达"进度如何"、"状态怎样"等意图

**决策**：
生成并展示项目进度报告

### 场景 5：用户想要修复问题
**条件**：
- 用户表达"修复bug"、"解决问题"等意图

**决策**：
1. 识别问题类型
2. 决定修复策略
3. 协调相应的 skills 进行修复
```

### 第三阶段：技能协调和执行

#### 1. 调用子技能
**执行模式**：
- **顺序执行**：按照标准流程依次执行 skills
- **跳步执行**：根据用户需求跳过某些阶段
- **回退执行**：返回到之前的阶段重新执行
- **并行执行**：同时执行多个独立的 skills（如果适用）

#### 2. 监控执行状态
**监控要点**：
- Skill 是否成功执行
- 是否产生了预期的输出
- 是否有错误或异常
- 用户是否对结果满意

#### 3. 处理执行结果
**成功情况**：
- 更新项目状态
- 记录完成时间
- 规划下一步行动

**失败情况**：
- 记录错误信息
- 分析失败原因
- 提供解决方案
- 更新阻塞因素

## 🎯 核心工作模式

### 模式 1：自动流程模式
**触发条件**：用户说"开始新版本"或"开始开发"

**执行流程**：
1. 调用 `version-designer` 设计版本
2. 用户确认后，依次调用后续 skills
3. 每个阶段完成后自动进入下一阶段
4. 直到完成所有阶段

**用户交互**：
- 在每个关键节点请求用户确认
- 允许用户暂停和继续
- 允许用户跳过某些阶段

### 模式 2：辅助决策模式
**触发条件**：用户问"下一步做什么"或"给我建议"

**执行流程**：
1. 分析当前项目状态
2. 判断下一步应该做什么
3. 提供多个选项供用户选择
4. 根据用户选择执行相应操作

**用户交互**：
- 展示当前状态
- 提供可行的选项
- 解释每个选项的利弊
- 等待用户决策

### 模式 3：状态报告模式
**触发条件**：用户问"进度如何"或"当前状态"

**执行流程**：
1. 读取项目状态文件
2. 生成详细的进度报告
3. 展示已完成和待完成的任务
4. 提供下一步建议

**用户交互**：
- 展示清晰的状态报告
- 突出显示重要信息
- 提供进度可视化

### 模式 4：问题修复模式
**触发条件**：用户报告问题或错误

**执行流程**：
1. 理解问题类型和严重程度
2. 决定修复策略
3. 协调相应的 skills 进行修复
4. 验证修复结果
5. 更新项目状态

**用户交互**：
- 详细记录问题描述
- 提供修复方案
- 执行修复操作
- 确认问题解决

## 🚨 常见问题处理

### 问题 1：项目状态文件损坏
**现象**：`.project-state.json` 文件损坏或丢失
**解决**：
- 从 `.project-log.md` 重建状态
- 扫描现有文件推断状态
- 请求用户确认重建的状态

### 问题 2：技能执行失败
**现象**：某个 skill 执行失败
**解决**：
- 记录详细的错误信息
- 分析失败原因
- 提供替代方案
- 允许用户跳过或重试

### 问题 3：前置条件不满足
**现象**：某个阶段的前置条件不满足
**解决**：
- 明确指出缺失的前置条件
- 提供完成前置条件的步骤
- 允许用户手动完成后再继续
- 或根据用户选择跳过

### 问题 4：用户需求变更
**现象**：在执行过程中用户改变需求
**解决**：
- 暂停当前流程
- 评估变更影响
- 更新项目状态
- 重新规划流程
- 请求用户确认新流程
