# 版本设计器

这个技能帮助你设计项目的新最小功能版本，通过分析现有项目架构、历史决策和评审结果，评估新业务需求的兼容性，并生成完整的版本设计文档。

## 🎯 技能概述

### 主要功能
- 📖 **项目背景分析**：读取 `business/global` 了解项目总体设计
- 📜 **历史回顾**：分析 `docs/` 中的历史决策和评审结果
- 🎨 **兼容性评估**：评估新业务与现有架构的兼容性
- ⚠️ **冲突识别**：识别潜在的技术、架构和业务冲突
- 💡 **建议提供**：基于项目经验提供优化建议
- 📝 **文档生成**：生成完整的版本设计文档

### 生成的文档
新版本目录将包含：
- **需求.md**：用户故事、功能列表、非功能性需求、依赖项、风险
- **设计.md**：架构概览、模块划分、API 定义、数据流、安全设计、性能优化
- **数据库.md**：表结构设计、ER 图、索引策略、关系定义、迁移计划
- **版本说明.md**：版本概览、变更摘要、升级指南

### 前提条件
- ✅ `business/global/` 目录存在且包含项目总体设计文档
- ✅ `docs/` 目录存在且包含历史决策和评审记录
- ✅ 用户提供了新业务需求的清晰描述
- ✅ 项目已有至少一个版本（作为参考基础）

### 第二阶段：需求理解和澄清

#### 1. 收集新业务需求
与用户深入交流，理解：
- **业务背景**：为什么需要这个功能？
- **用户价值**：为用户解决什么问题？
- **核心功能**：具体要实现什么？
- **验收标准**：如何判断功能完成？
- **优先级**：MUST/SHOULD/NICE TO HAVE
- **约束条件**：时间、资源、技术限制

#### 2. 需求澄清问题
```

4. 核心的用户操作流程是什么？
5. 需要存储哪些数据？
6. 是否需要与其他模块集成？
7. 有哪些性能或安全要求？
8. 期望的交付时间？

**参考格式**：
作为 [用户角色]，我需要 [功能]，以便 [价值/目标]
```

## API 接口定义

### RESTful 规范

#### 请求格式

```typescript
  // 字段定义
```

#### 响应格式

interface ResourceResponse {

### 错误处理

| HTTP 状态码 | 错误代码 | 说明 |
|-----------|---------|------|
| 400 | VALIDATION_ERROR | 请求参数验证失败 |
| 404 | NOT_FOUND | 资源不存在 |
