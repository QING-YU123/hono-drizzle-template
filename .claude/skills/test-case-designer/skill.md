# 测试用例设计器

这个技能帮助你根据 OpenAPI 文档设计全面的业务测试用例，确保覆盖功能正确性、异常处理、边界条件和数据一致性四个维度。

## 🎯 技能概述

### 主要功能
- 📖 **读取 OpenAPI 文档**：从 `OpenAPI/` 目录读取业务 API 规范
- 🎨 **分析 API 结构**：解析 API 端点、请求参数、响应格式
- ✏️ **设计测试用例**：根据测试设计原则生成详细测试用例
- 📋 **生成用例文档**：创建标准化的测试用例文档
- 🔄 **版本管理**：支持多版本测试用例管理

### 设计原则
测试用例设计遵循以下四个维度：
1. **功能正确性（Happy Path）**：验证核心业务操作在正常情况下能正确执行
2. **异常处理（Unhappy Path）**：测试输入验证、资源不存在、业务规则冲突等
3. **边界条件**：测试数值边界、字符串长度、特殊字符等极限情况
4. **数据一致性**：验证创建后能查询、更新后数据正确、删除后不再存在

### 前提条件
- ✅ 项目存在 OpenAPI 规范文档（位于 `OpenAPI/` 目录）
- ✅ OpenAPI 文档符合 3.0.x 规范
- ✅ 业务迭代目录结构存在（`business/iterations/`）

## 🔄 完整工作流程

### 第一阶段：环境分析和文档定位（必做）

#### 1. 扫描 OpenAPI 文档
```bash
# 扫描 OpenAPI 目录
Glob OpenAPI/*.json

# 检查是否存在 OpenAPI 文档
# 如果有多个文档，与用户确认使用哪个
```

**OpenAPI 文档命名规则：**
- 格式：`{module-name}.openapi.json`（如 `user.openapi.json`）
- 或 `{module-name}.openapi.yaml`（如果使用 YAML 格式）

#### 2. 扫描业务迭代版本
```bash
# 扫描业务迭代目录
Glob business/iterations/*

# 与用户确认当前版本
# 如果只有一个版本，自动使用该版本
# 如果有多个版本，询问用户选择或创建新版本
```

#### 3. 分析 OpenAPI 文档结构
```bash
# 读取 OpenAPI 文档
Read OpenAPI/{module-name}.openapi.json

# 解析以下内容：
- info: API 基本信息（标题、版本、描述）
- servers: 服务器 URL
- paths: API 端点和操作
- components.schemas: 数据模型定义
- tags: API 分组标签
```

**分析要点：**
- API 端点列表（path + method）
- 请求参数（path parameters、query parameters、request body）
- 响应格式（成功响应和错误响应）
- 数据验证规则（required、format、minimum、maximum 等）
- 业务约束（unique、nullable 等）

## 🎯 测试用例设计最佳实践

### 1. 功能正确性（Happy Path）
- 每个 API 操作至少有一个成功场景
- 验证返回数据的完整性和正确性
- 检查所有必需字段是否存在
- 验证业务逻辑的正确执行

### 2. 异常处理（Unhappy Path）
- **输入验证**：缺少必填字段、格式错误、类型错误、长度超出
- **资源不存在**：GET/PUT/DELETE 操作时的 404 场景
- **业务规则冲突**：唯一约束违反、状态冲突、权限不足
- **格式错误**：JSON 格式错误、参数类型错误

### 3. 边界条件
- **数值边界**：最小值、最大值、0、负数
- **字符串长度**：空字符串、最大长度、超长字符串
- **特殊字符**：HTML 标签、SQL 关键词、Unicode 字符
- **分页边界**：page=0、limit=0、超出最大页数

### 4. 数据一致性
- **创建后查询**：POST 创建的资源可以通过 GET 获取
- **更新后验证**：PUT 更新后，GET 返回更新后的值
- **删除后验证**：DELETE 后，GET 返回 404
- **级联影响**：验证关联资源的行为

### 5. 测试用例独立性
- 每个测试用例应能独立运行
- 不依赖其他用例的执行结果
- 测试数据应在用例内准备或使用 beforeAll
- 测试完成后应清理数据

## 🚨 常见问题处理

### 问题 1：找不到 OpenAPI 文档
**现象**：`OpenAPI/` 目录为空或不存在
**解决**：
- 检查 OpenAPI 文档是否已生成
- 使用 /openapi-generator 先生成 OpenAPI 文档
- 或提供 OpenAPI 文档的路径

### 问题 2：OpenAPI 文档格式不正确
**现象**：无法解析 OpenAPI 文档
**解决**：
- 检查文档是否符合 OpenAPI 3.0.x 规范
- 使用 OpenAPI 验证工具验证文档
- 修正文档格式错误

### 问题 3：版本目录不存在
**现象**：`business/iterations/` 目录为空
**解决**：
- 创建新的版本目录
- 或使用现有版本（如果有）

### 问题 4：测试用例保存失败
**现象**：无法写入测试用例文档
**解决**：
- 确认目录结构存在
- 检查文件写入权限
- 创建必要的目录结构

## 🎯 技能特点

### 与 test-generator 的配合
1. **test-case-designer** 负责设计测试用例（本文档）
   - 输入：OpenAPI 文档
   - 输出：测试用例文档
   - 作用：规划测试策略和测试场景

2. **test-generator** 负责生成测试代码
   - 输入：测试用例文档
   - 输出：Vitest 测试代码和测试报告
   - 作用：实现测试用例

**完整流程**：
```
OpenAPI 文档 → test-case-designer → 测试用例文档 → test-generator → 测试代码和报告
```
