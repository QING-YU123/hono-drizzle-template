# OpenAPI 文档生成器

这个技能帮助你生成和维护 OpenAPI 文档，基于项目现有的业务代码，自动生成符合标准的中文 API 文档。

## 📋 OpenAPI 编写标准（要点总结）

**⚠️ 重要：以下标准基于现有 OpenAPI/user.openapi.json 提取，必须严格遵守，无需再次阅读原文件。**

### 1. 文档结构规范

#### 基本信息
- **openapi 版本**：3.0.3
- **title**：中文模块名称，如 "Hono + Drizzle 用户管理 API"
- **version**：1.0.0
- **description**：详细描述，包含：
  - 功能概述
  - 系统特点列表（使用 **加粗** 标注）
  - 业务说明

#### 服务器配置
- **url**：http://localhost:3001
- **description**："本地开发环境"

### 2. 接口文档编写规范（关键要点）

#### 每个接口必须包含的描述格式：
```markdown
### 业务说明

[一段简要说明接口功能的文字]

#### 功能特点
- **特点1**：说明
- **特点2**：说明

#### 使用场景
- 场景1
- 场景2

#### 验证规则
- **参数名**：说明
- **参数名**：说明

#### 错误处理
- **状态码**：错误说明
- **状态码**：错误说明

#### 注意事项
- 注意点1
- 注意点2
```

#### 必需字段：
- **summary**：简短中文标题
- **description**：详细业务说明（按上述格式）
- **operationId**：驼峰命名（如 getAllUsers）
- **tags**：模块分类

#### 响应示例规范：
- 每个响应都要有中文 description
- **成功响应**：包含实际示例数据
- **错误响应**：包含所有可能的错误情况
- 使用中文示例名称（如 "创建成功"、"邮箱已存在"）

### 3. 数据模型规范

#### Schema 定义要点：
- **description**：详细的中文说明
- **example**：实际的示例值
- **required**：明确标注必填字段
- **nullable**：明确标注可空字段
- **additionalProperties**：设为 false 严格模式

#### 命名规范：
- 实体模型：User、Article、Order 等
- 创建请求：Create{Entity}Request
- 更新请求：Update{Entity}Request
- 删除响应：Delete{Entity}Response
- 错误响应：ErrorResponse

### 4. 示例数据规范

#### 示例要求：
- 使用真实的中文示例数据
- 使用中文姓名：张三、李四、王五
- 使用合理的邮箱：zhangsan@example.com
- 使用合理的日期格式：2024-01-15T08:30:00.000Z
- null 值明确标注为 null

## 🎯 工作模式

### 模式一：新建文档（create）

**使用场景：** 为新的业务模块创建 OpenAPI 文档

**工作流程：**
1. 分析指定 controller 文件的业务逻辑
2. 理解每个接口的功能和参数
3. 生成符合标准的完整 OpenAPI 文档

**适用情况：**
- 新建了 controller 文件
- 需要为现有模块补充 API 文档

### 模式二：修改文档（update）

**使用场景：** 更新现有 OpenAPI 文档

**工作流程：**
1. 读取指定的 OpenAPI 文件
2. 根据用户新增的业务内容更新文档
3. 保持原有格式和风格的一致性

**适用情况：**
- 接口功能发生变化
- 新增了接口
- 修改了参数或响应

## 🔄 完整工作流程

### 第一阶段：项目分析（Plan 模式）

#### 1. 分析业务代码
```bash
# 读取指定 controller 文件
Read src/controllers/[name].controller.ts

# 读取对应 service 文件
Read src/services/[name].service.ts

# - 接口路径和 HTTP 方法
# - 参数验证规则

# - 错误处理方式
# - 响应数据格式
```

#### 2. 确定模块边界
- **简单业务**：按 controller 文件边界划分
- **复杂业务**：即使涉及多表操作，仍按 controller 划分
- 一个 controller 对应一个 OpenAPI 文档

#### 3. 理解用户需求
向用户确认：
- 工作模式：新建还是修改？
- 目标文件：哪个 controller？哪个 OpenAPI 文件？
- 更新内容：如果是修改模式，具体要更新什么？

#### 4. 生成任务简报
```
任务简报：
- 工作模式：[新建/修改]
- 目标模块：[模块名称]
- 源文件：src/controllers/[name].controller.ts
- 目标文件：OpenAPI/[name].openapi.json
- 涉及接口：[列出所有接口]
- 主要变更：[说明将要生成/更新的内容]
```

**等待用户确认后再执行。**

### 第二阶段：文档生成/更新（Agent 模式）

#### 新建模式执行步骤：
1. 创建 OpenAPI 文档框架
2. 为每个接口添加详细说明
3. 定义所有数据模型
4. 添加完整示例
5. 确保符合所有规范

#### 修改模式执行步骤：
1. 读取现有 OpenAPI 文档
2. 定位需要修改的部分
3. 更新接口说明和数据模型
4. 保持格式和风格一致
5. 验证更新后的文档完整性

## 🔍 模块边界确定指南

### 简单业务模块
**特征：** 一个 controller 处理单一数据表的 CRUD 操作

**示例：**
- `user.controller.ts` → `user.openapi.json`
- `article.controller.ts` → `article.openapi.json`

**模块边界：** controller 文件边界

### 复杂业务模块
**特征：** 涉及多表操作，但仍在一个 controller 中

**示例：**
- `order.controller.ts` → 处理订单、订单项、支付等多表操作
- `product.controller.ts` → 处理商品、分类、库存等多表操作

**模块边界：** 仍按 controller 文件边界划分

### 特殊情况
**跨 controller 的复杂业务：**
- 如果业务逻辑分散在多个 controller 中
- 仍以主 controller 为准创建一个 OpenAPI 文档
- 在文档中说明相关联的其他接口
